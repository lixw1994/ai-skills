# ljg-fomo: 技术资讯研究 — 杀死 FOMO

一条命令完成：多平台信息采集 → 噪音过滤 → 结构化笔记 + 便携卡片。

设计哲学：FOMO 的本质不是信息不够，是**没有锚点**。这个 skill 的目标不是收集一切，而是用 10 分钟建立一个判断锚点——知道它是什么、为什么火、跟你有没有关系、该不该投入时间。

## 模式

**强制 NATIVE 模式。** 本 workflow 是多 skill 管道（Research → ContentAnalysis → ljg-card），不走 Algorithm 七步流程。

## 参数

| 参数 | 说明 | 示例 |
|------|------|------|
| 话题 | 必填，技术话题/关键词/链接 | `Deno 2`、`Claude Code`、一条 HN 链接 |
| `-d` | 深度模式，增加内容提炼步骤 | |
| `-vs {对比项}` | 对比模式，与已有方案对比 | `-vs Node.js` |

## 执行

### 1. 解析参数

从用户消息中提取技术话题。如果用户给的是链接，先 WebFetch 获取标题和上下文，确定话题关键词。

### 2. 全维度研究（Research extensive — 单次调用，12 个 Agent 并行）

调用 Skill tool 执行 `Research`，使用 extensive 模式。

**核心设计：同时采集"事实层"和"观点层"——一半 Agent 搞清楚它是什么，一半 Agent 搞清楚别人怎么看。**

**研究提纲（传入 Research 的 prompt）：**

```
对「{话题}」进行快速但全面的技术资讯研究。目标不是写综述，是帮一个忙碌的技术人在 10 分钟内建立判断锚点。

研究覆盖以下维度，每个维度需要中英文双语搜索：

**维度 A — 它是什么（What）**
一句话定义。解决什么问题？核心创新点或差异化在哪里？谁做的？什么阶段（论文/alpha/production-ready）？

**维度 B — 为什么现在火（Why Now）**
什么事件触发了关注？（发布、融资、大佬背书、争议事件、benchmark 刷榜）时间线是什么？是真的突破还是营销驱动？

**维度 C — 技术实质（How）**
核心技术原理或架构（简洁版）。与现有方案的关键差异。已知的局限和 trade-off。
{如有对比项：重点对比「{话题}」vs「{对比项}」，维度包括：性能、生态、学习曲线、适用场景、社区活跃度。}

**维度 D — 社区信号（Signal）**
Hacker News 上的讨论：主流观点和反对意见各是什么？
X/Twitter 上的讨论：谁在推、谁在黑、关键 thread 有哪些？
Reddit（r/programming, r/MachineLearning 等相关 subreddit）的讨论。
GitHub：star 数、contributor 数、issue 活跃度、最近 commit 频率。
筛选标准：
- 要：有信息增量的观点（技术分析、实际使用体验、有理有据的批评）
- 不要：纯转发无分析、营销内容、情绪宣泄
- 优先收集：核心开发者/领域权威的发言
返回内容标题/作者、URL、一句话摘要。

**维度 E — 实际采用（Adoption）**
谁在用？用在什么场景？有没有 production 案例？迁移成本多大？生态成熟度如何？

**维度 F — 深度内容发现**
搜索高质量的深度分析文章和视频：
- 技术博客（个人博客 > 公司博客）
- YouTube 技术讲解（优先 15 分钟以上）
- 中文社区：B站技术讲解、知乎深度回答、公众号技术分析
返回内容标题、URL、一句话摘要。
```

等待 Research 完成，获得全维度研究结果。

### 3. 内容提炼（ContentAnalysis — 深度模式）

**默认跳过此步。** 仅在用户指定 `-d` 深度模式时执行。

从步骤 2 返回的结果中，选择最有价值的 3-5 个 URL（优先选择：核心开发者博文、权威技术分析、关键争议 thread）。

**对每个 URL 并行启动 Agent subagent：**

每个 subagent 调用 Skill tool 执行 `ContentAnalysis`，传入 URL，使用 fast 深度级别，提取核心知识点。

**降级规则：**
- ContentAnalysis 失败则跳过，不阻塞
- 步骤 2 的研究结果已经足够生成文档

收集所有成功提炼的内容摘要。

### 4. 合成 Markdown 文档

将步骤 2（研究结果）和步骤 3（内容提炼，如有）合成为一份结构化 Markdown 文档。

**文档结构：**

```
---
title: "{话题}" 技术资讯研究
date: YYYY-MM-DD
tags: [tech-research, {领域标签}]
---

# {话题}

> **一句话判断**：{用一句大白话说清楚这个东西是什么、值不值得关注}

## 它是什么

{一段话解释清楚。不是 Wikipedia 式定义，是"如果你只有 30 秒跟朋友解释"的版本。}

**关键信息：**
- 谁做的：{团队/公司}
- 什么阶段：{论文 / alpha / beta / production-ready}
- 首次出现：{时间}
- 最近事件：{触发关注的事件}

## 为什么火

{时间线叙事。什么时候出现的、什么事件引爆了关注、是实质突破还是 hype cycle。}

## 技术核心

{简洁的技术原理。面向有技术背景但不在该领域的读者。}

**关键差异**（vs {对比项 or 该领域主流方案}）：
| 维度 | {话题} | {对比项 or 主流方案} |
|------|--------|-----------|
| ... | ... | ... |

> 无 `-vs` 时，Research 自行发现该领域最常被对比的方案；有 `-vs` 时，锁定用户指定的对比项。

**已知局限：**
- {局限 1}
- {局限 2}

## 社区声音

### 看好的观点
- {观点 1}（{来源/作者}）
- {观点 2}

### 质疑的观点
- {观点 1}（{来源/作者}）
- {观点 2}

### 关键讨论
- [{标题}]({URL}) — {一句话摘要}

## 采用现状

{谁在用、用在哪、生态怎么样。如果还太早就直说"还早"。}

## 我的判断

> **跟我有没有关系**：{根据用户的技术栈和关注领域，给出个性化判断}
>
> **建议行动**：
> - [ ] {具体行动 1，如"花 1 小时跑一下官方 tutorial"}
> - [ ] {具体行动 2，如"先观望，3 个月后再看生态"}
> - [ ] {或者"不用管，跟你的方向无关"}

## 深度阅读

**必读**（投入时间最高性价比的内容）：
- [{标题}]({URL}) — {为什么值得读}

**选读**（有时间再看）：
- [{标题}]({URL}) — {一句话摘要}

**视频**：
- [{标题}]({URL}) — {时长} | {一句话摘要}
```

**文件命名**：保存到 `{ProjectRootDir}/Notes/fomo/` 目录：
`YYYY-MM-DD-{话题}技术研究.md`

**写作要求**：
- 语气是给自己写的研究笔记，不是新闻稿
- "我的判断"部分要有态度，不要两边讨好
- 社区声音必须同时呈现支持和质疑，不许只挑好的
- 没有确切信息时直说"信息不足"，不编造
- 所有观点标注来源，所有链接可点击

### 5. 铸造便携卡片（ljg-card）

从步骤 4 的 Markdown 文档中提取核心内容，铸造卡片。

**卡片 A — 话题速览（信息图）：**

调用 Skill tool 执行 `ljg-card -i -o {ProjectRootDir}/Notes/fomo/`，输入内容为：一句话判断 + 关键信息 + 技术核心差异 + 社区信号摘要 + 建议行动。一张图看懂要不要花时间。

**卡片 B — 深度阅读清单（长图，仅 `-d` 深度模式）：**

调用 Skill tool 执行 `ljg-card -l -o {ProjectRootDir}/Notes/fomo/`，输入内容为：必读 + 选读 + 视频推荐列表，每条附一句话理由。手机上随时翻阅。

> 默认模式只铸卡片 A；深度模式两张卡片**并行**铸造。

### 6. 汇总报告

```
════ FOMO 已杀死 ═══════════════════════
🔬 话题: {话题}
📝 研究笔记: {Markdown 文件路径}
🖼️ 速览卡片: {PNG 文件路径}
🖼️ 阅读清单卡: {PNG 文件路径，仅深度模式}
📊 信息覆盖: {N}个平台 | {M}条关键讨论 | {K}篇深度内容
⏱️ 建议行动: {一句话概括}
```

## 关键约束

- 速度优先——默认跳过内容提炼步骤，10 分钟内出结果
- Research 阶段多 Agent 并行，同时覆盖事实层和观点层
- 社区声音**必须呈现对立观点**，不许只报喜不报忧
- "我的判断"部分有立场、可执行，不是各打五十大板
- 中英文双语搜索，英文社区（HN/Reddit/X）和中文社区（知乎/B站/公众号）同时覆盖
- 没有确切信息时宁可留空，不编造
