# 事实核查报告

对一个具体说法进行真伪判定。核心只做一件事：这个说法，到底站不站得住？

## 认知起点

事实核查不是"我觉得是真的/假的"，是拿证据称重量。

三条铁律：
- 谁主张谁举证——说法本身不是证据，支撑说法的东西才是
- 一个信源不算数——至少两条独立路径指向同一个结论才算初步验证
- 事实、推断、观点三件事绝不混淆——发生了什么是事实，基于事实的推理是推断，带立场的解读是观点

## 输入

一条具体的说法、新闻标题、社交媒体截图、转发消息、引用的"数据"、坊间传闻，或任何需要判定真伪的内容。给出原文即可——用 WebSearch 和 WebFetch 追溯和验证。

## 报告结构

事实核查报告短平快，直接给结论，然后展开证据。不写论文，写判决书。

### 待核查说法

原样引用用户提供的说法。不改写、不概括——改写本身就会引入偏差。

如果说法包含多个可独立核查的断言，拆开编号：

1. 断言 A：……
2. 断言 B：……
3. 断言 C：……

### 判定结论

先给结论，再展开。每个断言给出判定：

| # | 断言摘要 | 判定 | 信心度 |
|---|----------|------|--------|
| 1 |          |      |        |
| 2 |          |      |        |

**判定等级**（六级）：

- ✅ **证实** — 多个独立信源一致，有硬证据，无实质性反驳
- 🟢 **大致属实** — 核心为真，部分细节有出入或无法完全验证
- 🟡 **部分属实** — 真假混杂，需要拆开说——哪部分真、哪部分假
- ⚪ **无法证实** — 证据不足，既不能证实也不能证伪
- 🟠 **误导性** — 个别事实可能为真，但拼接方式制造了错误印象
- ❌ **证伪** — 有确切证据证明说法不成立

**信心度**：高 / 中 / 低——取决于证据的数量和质量，不是取决于我多自信。

### 证据展开

对每个断言，展开核查过程：

**断言 #N：[原文]**

**信源追溯**：
这个说法最早出自哪里？从源头到用户看到的版本，经过了几手？每一手变了什么？

**核查证据**：

| 证据       | 来源             | 支持/反驳/无关 | 证据强度   |
|------------|------------------|----------------|------------|
| 证据 1     |                  |                | 硬/软/弱   |
| 证据 2     |                  |                |            |

证据强度定义：
- **硬证据**：官方文件、原始数据、法庭记录、学术论文、当事人一手陈述
- **软证据**：严肃媒体报道、专家分析、二手但可追溯的引用
- **弱证据**：匿名信源、自媒体、社交媒体转发、"据说"

**矛盾与疑点**：
如果证据之间有矛盾，矛盾在哪？哪个版本有更硬的证据？如果没有矛盾，是因为确实一致，还是因为所有信源都在引用同一个源头（伪多源）？

**判定理由**：
一段话说清楚为什么给出这个判定。指向具体证据，不说"综合考虑"。

### 传播分析（如适用）

如果这个说法有广泛传播，简要分析：
- 它为什么会传播？——利用了什么情绪、偏见或信息真空？
- 传播过程中变形了吗？——最早版本和现在流传的版本有什么不同？
- 有没有被刻意放大的迹象？——水军、营销号、利益相关方推动？

这一节只在说法已形成广泛传播时写，不是每次都要分析。

### 一句话总结

用一句不超过 50 字的话给出最终结论。这句话要能直接转发给问你"这是真的吗"的朋友。

## 输出

- 格式：Markdown
- 目录：`{ProjectRootDir}/Notes/fact-check/`
- 命名：`YYYY-MM-DD-核查-CLAIM_SUMMARY.md`
  - 例：`2026-04-07-核查-某某新闻真伪.md`
- 用 Write 工具写入，写完告知完整路径

## 生成规则

1. 基于可查证的信息，不编造。查不到的标注"未找到可靠信源"，判"无法证实"
2. 结论先行。读者最想知道的是真还是假，不要让他们翻到最后才看到
3. 每个判断附具体信源——出处、时间、作者，不是"有报道称"
4. 区分事实、推断和观点，三者绝不混淆
5. 不做假平衡。不是每个说法都"有待商榷"——证据够了就判，不够就说不够
6. 禁止出现：众所周知、不言而喻、据说、网传（不标来源的）
7. 简洁为上。核查报告不是调研报告——说清楚就停，不展开背景知识
8. 中文撰写
