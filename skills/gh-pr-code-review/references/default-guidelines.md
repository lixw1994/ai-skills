# Code Review 默认检查清单

本文档提供语言无关的代码审查检查清单。Sub Agent 和主 Agent 在执行 Review 时参考此清单，并结合 `.code-review.json` 中的项目特定规则。

## 1. 命名与拼写

- [ ] 变量名是否清晰表达其用途，避免单字母变量（循环变量 i/j/k 除外）
- [ ] 函数/方法名是否准确描述其行为
- [ ] 是否有拼写错误（常见：recieve→receive, seperate→separate, occured→occurred）
- [ ] 命名是否符合项目配置的命名规范（参考 `.code-review.json` 中的 `namingConventions`）
- [ ] 布尔变量是否使用 is/has/can/should 等前缀
- [ ] 集合变量是否使用复数形式
- [ ] 避免使用非通用缩写

## 2. 设计与可读性

- [ ] 函数是否遵循单一职责原则
- [ ] 函数长度是否合理（建议不超过 50 行，超过应考虑拆分）
- [ ] 参数数量是否过多（建议不超过 5 个，超过应考虑用对象封装）
- [ ] 嵌套层级是否过深（建议不超过 3 层，超过应考虑提前返回或提取函数）
- [ ] 复杂逻辑是否有注释说明
- [ ] 魔法数字/字符串是否定义为常量
- [ ] 是否有明显的重复代码可以提取
- [ ] 条件判断是否易于理解（考虑正向条件、卫语句）

## 3. 错误处理与边界条件

### 3.1 空值与类型安全
- [ ] 参数是否进行了必要的空值检查
- [ ] 集合操作前是否检查了 null/undefined 或空集合
- [ ] 类型转换是否安全，是否有溢出风险
- [ ] 可选值是否在使用前做了存在性检查

### 3.2 集合与边界
- [ ] 数组/列表索引访问是否有边界检查
- [ ] 循环条件是否正确（注意 off-by-one 错误）
- [ ] 空集合场景是否被正确处理

### 3.3 错误处理
- [ ] 异常/错误是否被恰当捕获和处理
- [ ] 是否避免了吞掉异常（catch 后无任何处理）
- [ ] 错误信息是否有足够的上下文便于排查
- [ ] 是否有适当的错误日志

### 3.4 资源管理
- [ ] 打开的资源（文件、连接、流）是否正确关闭/释放
- [ ] 是否有资源泄漏风险

### 3.5 并发安全
- [ ] 共享状态是否有适当的同步机制
- [ ] 是否有竞态条件风险
- [ ] 是否有死锁风险

## 4. 安全性

- [ ] 是否有 SQL 注入风险（拼接 SQL 而非参数化查询）
- [ ] 是否有 XSS 风险（未转义的用户输入直接渲染）
- [ ] 是否有敏感信息硬编码（密码、Token、密钥）
- [ ] 外部输入是否经过校验和清洗
- [ ] 是否有路径遍历风险
- [ ] API 接口是否有适当的认证和授权检查

## 5. 性能

- [ ] 是否有明显的性能问题（如循环中重复创建对象、重复计算）
- [ ] 是否有 N+1 查询问题（循环中逐条查询数据库）
- [ ] 大数据集处理是否合理（是否需要分页、流式处理）
- [ ] 字符串拼接是否应使用更高效的方式（StringBuilder 等）

## 6. 代码变更意图

- [ ] 是否存在可能是"不小心"改变原有逻辑的情况
- [ ] 删除的代码是否确实不再需要
- [ ] 条件判断是否被意外修改
- [ ] return/break/continue 语句位置是否被意外改变

---

## 优先级定义

| 优先级 | 标记 | 含义 | 示例 |
|--------|------|------|------|
| 高 | 🔴 | Bug 或安全漏洞，会导致运行时错误或安全问题 | SQL 注入、空指针、资源泄漏 |
| 中 | 🟡 | 设计问题或边界条件，可能在特定场景下出问题 | 缺少错误处理、并发问题、N+1 查询 |
| 低 | 🟢 | 命名或可读性问题，影响代码质量但不影响功能 | 命名不清晰、魔法数字、过长函数 |

## 评论原则

1. **只报告实质性问题**，不吹毛求疵
2. **必须给出具体建议**，不要只说"这里有问题"
3. **结合上下文判断**，不要脱离实际使用场景
4. **尊重作者意图**，区分"bug"和"不同的实现选择"
5. **对不确定的问题用疑问句**，如"这里是否考虑了…的场景？"
