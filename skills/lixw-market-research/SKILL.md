# 市场调研报告

从市场营销的角度生成一份产品可行性分析。核心只问一个问题：这群人的痛够不够真，你的止痛方式够不够独特，账算不算得过来？

## 认知起点

市场不是一个数字，是一群人正在忍受的痛苦和他们愿意为止痛付的代价。

所以不量面积，看深度：
- 不问"市场有多大"，问"谁在痛、痛到什么程度、痛了多久"
- 不问"竞争对手是谁"，问"用户现在用什么凑合的方案忍着"
- 不问"能不能赚钱"，问"用户愿意花多少钱换掉现在的痛苦"

## 输入

产品创意、商业构想、功能描述、竞品链接、用户反馈、行业报告，或任何描述一个产品的材料。已上线的产品给名字即可——用 Research skill 或 WebSearch 搜集最新数据。

## 报告结构

以下七个区块是骨架，不是填空题。哪个区块最能帮判断，那个就多写；信息不足的一两句带过。报告为决策服务，不为完整性服务。

### 一、这个痛是什么

一张表 + 一段需求诊断。

| 维度     | 内容                                         |
|----------|----------------------------------------------|
| 产品定义 | 用一句话说清楚它解决什么问题                 |
| 目标用户 | 谁最痛？具体画像，不要"25-35岁白领"         |
| 痛点等级 | 止痛药 / 维生素 / 安慰剂（见下方定义）       |
| 付费意愿 | 有信号 / 待验证 / 无信号                     |
| 需求频次 | 每天用 / 每周用 / 偶尔用 / 一次性            |

痛点等级定义：
- **止痛药**：用户明确知道自己痛，正在花钱或花时间解决，你的方案能显著减少痛感。这是最好的生意。
- **维生素**：用户隐约觉得可以更好，但不解决也能活。需要教育成本，但一旦养成习惯有粘性。
- **安慰剂**：只有你觉得用户需要，用户自己不觉得。危险信号。

穿透表面看需求。"我需要一个项目管理工具"是表面，"我的团队每周花 8 小时在微信群里同步进度，每次都有信息遗漏"是真需求——后者告诉你痛有多深、现在在怎么忍。

### 二、现在怎么忍的

不做竞品列表，做替代方案地图。用户今天不用你的产品，他们用什么活着？

**现有方案矩阵**：

| 替代方案     | 用户怎么用        | 好在哪           | 差在哪           | 转换成本         |
|--------------|--------------------|------------------|------------------|------------------|
| 方案 A       |                    |                  |                  |                  |
| 方案 B       |                    |                  |                  |                  |
| 什么都不做   |                    |                  |                  |                  |

"什么都不做"也是竞争对手——而且往往是最强的那个。如果用户忍得了，你的产品就没戏。

**关键问题**：
- 用户对现有方案的不满有多强烈？是"有点烦"还是"忍无可忍"？
- 转换到你的方案，用户要付出什么代价（学习成本、数据迁移、习惯改变）？
- 这个代价和减少的痛苦相比，划不划算？

### 三、你凭什么止得了

差异化不是"我们有更好的用户体验"——这不是差异化，是自我感觉良好。

**真壁垒检验**：

回答以下问题，每个都要有具体事实支撑：

**独特价值在哪？**
你的方案解决了什么现有方案解决不了的问题？或者同样的问题你解决得好在哪，好多少？"好一点点"不够——用户的迁移惰性远比你想象的大，改善必须大到用户一眼就能感知。

**护城河是什么？**
- 网络效应：用的人越多越好用？（如果是，冷启动怎么解决？）
- 数据壁垒：用得越久数据越多，产品越好？（数据可迁移吗？）
- 品牌心智：提到这个品类就想到你？（你是第几个进场的？）
- 成本优势：同样的事你做得更便宜？（为什么？可持续吗？）
- 技术壁垒：别人做不到你能做到？（能维持多久？）

如果以上都没有——这不是产品，是功能。功能会被平台吞掉。

**定位声明**：
对于 [目标用户]，在 [使用场景] 中遇到 [核心痛点] 时，[产品名] 是 [品类] 中唯一能 [独特价值] 的选择，因为 [证据]。

如果这句话写不出来或写出来很牵强，产品定位有问题。

### 四、时机对不对

好产品在错误的时间做，和没做一样。这一节判断"为什么是现在"。

**窗口检测**：

什么变化让这个产品此刻成为可能或必要？通常是以下几种力量中的一种或几种同时发力：

- **技术拐点**：某项技术刚刚成熟到"够用但还没普及"——成本降到临界点、性能过了可用线、基础设施刚刚就位。太早你在替行业交学费，太晚你在红海里游泳。
- **行为迁移**：用户习惯正在发生不可逆的迁移——从线下到线上、从 PC 到移动、从搜索到推荐、从拥有到订阅。你的产品是顺着这股迁移做的，还是逆着的？
- **政策/法规变化**：新规打开了市场、旧规被废除释放了需求、合规要求创造了刚需。
- **成本坍缩**：某种曾经昂贵的东西突然变便宜了——算力、带宽、人才、内容生产、获客渠道。你在搭这趟便车吗？

**先行者 vs 后发者**：

这个品类有多成熟？

| 阶段 | 特征 | 做的意义 |
|------|------|----------|
| 无人区 | 没人在做，用户也不知道自己需要 | 教育成本极高，但赢了通吃 |
| 探索期 | 有几个先行者在试，尚无标准答案 | 可以借鉴避坑，还有定义品类的机会 |
| 增长期 | 领头羊出现，用户开始主动搜索 | 差异化切入，找大玩家没覆盖的缝隙 |
| 成熟期 | 格局已定，用户心智被占 | 除非有颠覆性不同，否则不碰 |

**时机判定**：
现在进场是早了、晚了、还是刚好？如果早了，等什么信号再动？如果晚了，还有没有差异化空间？

### 五、账算不算得过来

这一节做 ROI 推演——不追求精确（早期产品没法精确），追求量级正确。

**单位经济模型**：

| 指标 | 数值 | 计算依据 |
|------|------|----------|
| 客户获取成本（CAC） | | 获客渠道 × 转化率 |
| 客户生命周期价值（LTV） | | 客单价 × 复购频次 × 留存周期 |
| LTV / CAC | | >3 健康，1-3 勉强，<1 烧钱 |
| 回本周期 | | 几个月收回 CAC |
| 毛利率 | | |

没有精确数据时，用行业基准或类比估算，标注假设条件。

**增长引擎判断**：

产品靠什么方式获客？三种引擎，通常只能选一个主力：

- **病毒引擎**：用户用了之后会拉其他人来？NPS 高不高？分享动机是什么——利益驱动、社交货币、还是产品本身就需要多人使用？
- **粘性引擎**：留存率够高，自然增长大于流失？用户用了之后不想走？为什么不想走？
- **付费引擎**：花钱买用户，LTV > CAC？获客渠道有哪些？成本可控吗？竞争对手在抢同样的渠道吗？

**ROI 推演**：

给出三个场景下的 12 个月 ROI 估算：

| 场景 | 假设条件 | 投入估算 | 产出估算 | ROI |
|------|----------|----------|----------|-----|
| 乐观 | | | | |
| 基准 | | | | |
| 悲观 | | | | |

每个假设都要标清楚。ROI 不精确没关系，量级对就行——知道是 2 倍还是 0.5 倍比知道是 2.3 倍还是 2.7 倍重要得多。

### 六、外部视角校准

前面五节都是"从里往外看"——站在产品创造者的角度分析需求、竞争、差异化、时机和经济模型。这一节换一双眼睛，"从外往里看"，像一个聪明但刻薄的局外人那样，系统性地质疑前面所有分析中最脆弱的环节。

这一节的目的不是否定，是校准。很多产品死于创始人的自我说服——痛点是自己想象的、差异化是自己觉得的、ROI 是自己算的。外部视角的价值在于：把你从"我相信"拉回到"凭什么信"。

**对需求的质疑：**
回到第一节的痛点判断。如果你判断为"止痛药"——有没有可能你高估了痛的程度？用户说"我很需要"和用户真的掏钱之间有多大的距离？有没有类似的产品做了之后发现需求根本不存在的前车之鉴？

**对差异化的质疑：**
回到第三节。你说的差异化，用户能感知到吗？还是只有你自己觉得不同？如果你把产品描述念给一个目标用户听（不给演示），他能说出你和现有方案有什么不一样吗？如果领先的竞争对手花三个月抄你的核心功能，你还剩什么？

**对 ROI 的质疑：**
回到第五节。翻看你的假设条件——哪个假设最可能是错的？如果 CAC 翻倍、转化率砍半（这在早期产品中极为常见），模型还成立吗？有没有隐藏成本你没算进去——团队磨合、方向调整、功能返工、用户教育？

**对时机的质疑：**
回到第四节。你说"窗口打开了"——有没有可能是一个假窗口？市场上之所以没人做，是不是因为之前有人试过然后死了？如果这个时机真的好，为什么资源比你多的团队没有在做？

**综合校准：**
经过以上质疑，前面的判断需要修正吗？如果需要，在这里明确说出来：哪个判断从 A 调整为 B，为什么。如果质疑后依然成立，说明理由。不允许跳过这一步直接进入决策——每一条质疑都必须正面回应。

### 七、做不做

**结论**：做 / 小规模验证 / 不做

**如果做**：
- 最小可验证版本（MVP）是什么？砍到不能再砍，只保留验证核心假设的功能
- 第一批用户从哪来？不是"通过社交媒体推广"——要具体到哪个社群、哪类人、用什么钩子
- 验证周期多长？什么指标证明假设成立？

**如果小规模验证**：
- 验证什么假设？怎么验证？
- 花多少钱、多少时间验证？
- 什么结果说明该继续？什么结果说明该停？

**如果不做**：
- 核心原因是什么？（需求不真 / 差异化不够 / ROI 不划算 / 时机不对）
- 有没有变体值得考虑？（换个用户群？换个切入点？等某个条件成熟？）

**核心假设清单**：
列出这个决策依赖的 3-5 个关键假设，按风险从高到低排序。每个假设附一个"死亡信号"——什么数据出现说明假设错了。

### 最后一句

用一句话回答：这个产品解决的是真痛还是假痛？你的止痛方式有没有不可替代性？

## 生成规则

1. 基于真实信息和数据，不编造。信息不足的直接标注"待验证"，不硬撑
2. 敢判断。"需要进一步调研"是推卸责任——基于现有信息给出你的判断
3. 每个判断附证据——数据、用户行为、市场信号、具体事实
4. 禁止出现：蓝海市场、刚需痛点、千亿市场、风口赛道、赋能、抓手
5. 以说透为准，不计字数。三千字说清就三千字，需要八千字就八千字
6. 中文撰写
